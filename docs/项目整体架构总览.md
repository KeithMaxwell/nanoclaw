# NanoClaw 项目整体架构总览（基于源码）

> 目标：从代码层面给出 NanoClaw 的整体架构、关键模块职责、核心数据流与设计思想，帮助快速建立“全局心智模型”。

## 1. 一句话概述

NanoClaw 是一个“单进程编排 + 容器隔离执行”的个人 AI 助手系统：宿主 Node.js 进程负责消息接入、状态管理、调度与路由，真正的智能体执行（Claude Agent SDK）放在隔离容器中完成，从而在保持代码量小的同时获得较强安全性。

---

## 2. 代码结构总览

### 2.1 顶层目录角色

- `src/`：核心运行时（编排器、数据库、IPC、任务调度、容器调用、安全策略）
- `setup/`：安装/环境检测/服务化部署流程（分步骤 CLI）
- `container/`：智能体容器镜像与容器内 runner（含 MCP 工具桥接）
- `docs/`：需求、规范、安全、架构文档
- `groups/`：群组隔离工作区（每个群有自己的记忆与文件）
- `store/`（运行时生成）：SQLite 数据、认证等持久化状态
- `scripts/`：辅助脚本（如迁移）

### 2.2 src 目录模块分层

1. **编排层**
   - `src/index.ts`：主循环与全局编排入口
2. **消息与路由层**
   - `src/channels/*`：渠道注册机制
   - `src/router.ts`：上下行消息格式化与路由
3. **执行层（容器）**
   - `src/container-runner.ts`：容器生命周期、流式输出、IPC 快照
   - `src/container-runtime.ts`：运行时探测与清理
4. **状态层（持久化）**
   - `src/db.ts`：SQLite schema 与所有数据访问
5. **异步任务层**
   - `src/task-scheduler.ts`：cron/interval/once 调度
   - `src/group-queue.ts`：按群串行 + 全局并发上限队列
   - `src/ipc.ts`：容器与宿主之间任务/消息 IPC 目录监听
6. **安全与治理层**
   - `src/mount-security.ts`：额外挂载白名单校验
   - `src/sender-allowlist.ts`：消息触发与丢弃策略
   - `src/credential-proxy.ts`：凭据代理（不把密钥直接暴露给容器）

## 3. 核心运行流程（端到端）

## 3.1 启动阶段

1. 初始化数据库与迁移（`initDatabase`）
2. 加载路由状态、会话映射、已注册群组
3. 启动容器运行时检查与孤儿资源清理
4. 启动凭据代理（供容器侧按需取凭据）
5. 启动 IPC watcher 与 scheduler loop
6. 加载 channel registry 并连接可用渠道

## 3.2 消息进入系统

1. Channel 收到消息，通过回调进入主进程
2. 消息写入 DB（聊天元信息 + 消息）
3. 进入 `GroupQueue`：
   - 同一群串行，避免上下文竞争
   - 不同群受 `MAX_CONCURRENT_CONTAINERS` 并发限制

## 3.3 触发智能体执行

1. 编排器按群读取“上次 agent 游标之后”的消息
2. 触发规则（@助手名 / 主群免触发 / allowlist）通过后，格式化为 XML-like context
3. `container-runner` 启动或复用该群容器，向 agent runner 写入 prompt
4. 容器内 Claude Agent SDK 运行，结果流式回传

## 3.4 输出回传与状态落库

1. 流式结果经过 `formatOutbound` 清理后路由到对应 channel
2. 更新 `last_agent_timestamp`、session、task 运行日志
3. 若容器空闲超时则关闭 stdin 或回收

## 4. 数据模型（SQLite）

核心表在 `src/db.ts` 中创建：

- `chats`：会话元信息（jid、名称、渠道、是否群）
- `messages`：消息正文（复合主键 id + chat_jid）
- `scheduled_tasks`：计划任务定义（cron/interval/once + 状态）
- `task_run_logs`：任务执行历史
- `router_state`：游标/全局状态
- `sessions`：group_folder -> session_id
- `registered_groups`：被纳管群组配置（folder、trigger、containerConfig 等）

迁移策略采用“启动时轻量 ALTER TABLE + 回填”。

## 5. 关键设计点

## 5.1 单进程控制面 + 容器数据面

- 控制面（Node）保持简单：可理解、可修改
- 数据面（Agent 执行）放入容器：隔离文件系统与命令执行风险

## 5.2 群组隔离模型

每个群对应独立 folder、session、消息游标，避免上下文串扰；队列层保证“群内串行、群间并发”。

## 5.3 Channel 可插拔（技能化）

`src/channels/registry.ts` 采用工厂注册表；渠道能力通过“技能改代码”的方式加入，而非复杂插件框架。

## 5.4 安全边界前置

- mount allowlist 文件放在项目外（`~/.config/nanoclaw`）
- sender allowlist 可配置“仅限制触发”或“直接丢弃消息”
- 凭据经代理获取，减少容器直接接触宿主密钥

## 5.5 IPC 解耦

容器与宿主通过文件 IPC 目录交互（tasks/messages 快照与请求文件），降低耦合并增强可观察性。

---

## 6. setup 子系统（部署与可运维性）

`setup/index.ts` 用 `--step` 分步执行：

- `environment`：探测平台、Docker/Apple Container、已有状态
- `container`：构建并 smoke test 容器镜像
- `groups`/`register`：群组与注册流程
- `mounts`：挂载配置
- `service`：系统服务接入（launchd/systemd）
- `verify`：整体验证

每步通过 `emitStatus` 输出结构化结果，便于技能或自动化流程解析。

---

## 7. container 子系统（容器内执行）

- `container/Dockerfile`：构建 agent 运行镜像
- `container/agent-runner/src/index.ts`：容器内入口，接收 prompt 并调用 Claude Agent SDK
- `container/agent-runner/src/ipc-mcp-stdio.ts`：将任务管理/群组管理等能力封装为 MCP tools，经文件 IPC 与宿主协作

它的职责是“让 agent 在隔离环境中拥有受控工具能力”，而不是替代宿主编排器。

## 8. 测试覆盖现状

项目有较完整的 Vitest 测试集，覆盖：

- DB schema/迁移/查询
- GroupQueue 并发与重试
- IPC 授权与任务流

说明该项目虽然强调“轻量”，但对关键机制具备工程化回归保障。

## 9. 这个项目的核心思想（结论）

如果压缩成一句话：

**用最少、最直观的宿主代码做编排，把高风险能力放进真实容器隔离里，再用“按需改代码（skills）”替代“大而全配置系统”。**

可拆为三层理念：

1. **可理解优先**：代码库小、单进程、路径短，个人也能掌控。  
2. **安全边界优先**：不是靠“约定权限”，而是靠容器隔离与挂载白名单。  
3. **定制优先**：不追求默认支持一切，而是让你在自己 fork 上精确塑形。
